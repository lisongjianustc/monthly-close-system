## 二、安全漏洞核实

### 2.1 [Critical] Aviator 表达式未开启沙箱 — **属实**

**评估团队指证**: `RuleEngineServiceImpl.java` 依然使用默认的 `AviatorEvaluator.compile(expr)`，未配置沙箱。

**源码核实**:
```java
// RuleEngineServiceImpl.java:102-105
Expression compiledExpr = expressionCache.computeIfAbsent(
    rule.getExpression(),
    expr -> AviatorEvaluator.compile(expr)
);
```
确认无任何 `Options.ENABLE_PROPERTY_ACCESS` / `ENABLE_METHOD_ACCESS` 配置。

**判断**: **属实**。修复方案在CODE_REVIEW.md中被列为P3，尚未实施。风险边界：Aviator 5.3.3默认已禁用`Runtime.exec`等危险调用，但非白名单内的反射调用理论上仍有风险。

---

### 2.2 [High] 硬编码 JWT 密钥 — **部分属实（文件名有误）**

**评估团队指证**: `McConstants.java` 仍将 `JWT_SECRET` 以明文硬编码在代码中。

**源码核实**:
- `McConstants.java` 文件在项目中**不存在**
- 实际硬编码位置: `mc-common/src/main/java/com/mc/common/util/JwtUtil.java:18-19`
```java
private static final SecretKey KEY = Keys.hmacShaKeyFor(
    "mc-monthly-close-system-secret-key-2024-very-long-key-for-hs256".getBytes(StandardCharsets.UTF_8)
);
```

**判断**: **部分属实**。文件路径有误，但问题本身确实存在。建议通过 `@Value("${jwt.secret}")` 外置。

### 2.3 [Medium] 用户ID默认回退到管理员 — **部分属实（风险已被JWT过滤器消除）**

**评估团队指证**: `ImportController.java` 仍使用 `userId != null ? userId : 1L` 回退至管理员。

**源码核实**:
```java
// ImportController.java:59-60
userId != null ? userId : 1L,
userName != null ? userName : "系统管理员"
```
此代码确实存在。

**判断**: **部分属实**。代码仍存在，但JWT认证过滤器已强制要求认证，无有效token的请求在到达Controller之前即被拦截为401。此回退逻辑在当前认证体系下已无实际触发路径，但作为防御性编程仍建议修复。

## 三、逻辑缺陷核实

### 3.1 [Medium] 事务内try-catch吞异常 — **不属实（描述不精确）**

**评估团队指证**: `ImportServiceImpl.java` 在 `@Transactional` 方法中try-catch数据库异常，连接被标记为rollback-only，整批回滚。

**源码核实**:
- `doInsertBatch` (lines 105-139) 是 `@Transactional` 方法
- `flushBatch` (lines 141-150) 中确实存在 `try-catch { recordMapper.insert } catch { recordMapper.insert(status=INVALID) }`
- 但失败记录被重新 `insert` 为 `INVALID` 状态后，循环继续处理后续记录
- 最终 `batch.setStatus(failCount > 0 ? "PARTIAL_SUCCESS" : "COMPLETED")` — 外层事务**正常提交**

**判断**: **不属实**。原评估描述的问题行为"整批回滚"与实际代码行为不符。实际行为是：失败记录以INVALID状态入库，成功记录正常提交，非"整批回滚"。风险真实存在（事务边界设计不佳），但原描述的症状不准确。

## 四、性能问题核实

### 4.1 [High] 循环内单条插入(N+1) — **属实（修复不彻底）**

**评估团队指证**: `flushBatch` 方法内仍是 `for` 循环单条 `recordMapper.insert()`。

**源码核实**:
```java
// ImportServiceImpl.java:141-150
private void flushBatch(List<ImportRecord> records) {
    for (ImportRecord record : records) {
        try {
            recordMapper.insert(record);  // 每条一次INSERT
        } catch (Exception e) {
            record.setStatus("INVALID");
            record.setErrorMsg(e.getMessage());
            recordMapper.insert(record);  // 失败记录也是逐条INSERT
        }
    }
}
```

**判断**: **属实**。BATCH_SIZE=1000仅控制内存中攒批的阈值，每次 flushBatch 内部依然是逐条INSERT。万条记录仍有约2万次网络往返，性能问题修复不彻底。

### 4.3 [Medium] 魔法字符串与ObjectMapper重复实例化 — **属实**

## 六、新增/修正问题

### 6.1 [High] flushBatch中异常处理逻辑错误

**文件**: `ImportServiceImpl.java:141-150`

```java
private void flushBatch(List<ImportRecord> records) {
    for (ImportRecord record : records) {
        try {
            recordMapper.insert(record);
        } catch (Exception e) {
            record.setStatus("INVALID");
            record.setErrorMsg(e.getMessage());
            recordMapper.insert(record);  // ← 这里insert抛异常会直接跳出循环
        }
    }
}
```

**问题**: 当重试insert也抛出异常时，`flushBatch` 方法会直接中断，剩余记录全部丢失。

**修复建议**:
```java
private void flushBatch(List<ImportRecord> records) {
    List<ImportRecord> allRecords = new ArrayList<>(records);
    records.clear();
    for (ImportRecord record : allRecords) {
        try {
            recordMapper.insert(record);
        } catch (Exception e) {
            record.setStatus("INVALID");
            record.setErrorMsg(e.getMessage());
            try {
                recordMapper.insert(record);
            } catch (Exception ignored) {
                // 二次失败无法入内核，直接跳过
            }
        }
    }
}
```

### 6.2 [Medium] validateData逐条updateById

**文件**: `ImportServiceImpl.java:200-220`

```java
for (ImportRecord record : records) {
    recordMapper.updateById(record);  // N+1
}
```

**问题**: 与 `flushBatch` 相同，校验后逐条更新，万条记录产生N次网络往返。

**修复建议**: 使用 `updateBatchById` 批量更新。
